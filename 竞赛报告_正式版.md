# MoE Block 长序列训练显存优化 — 竞赛报告

## 执行摘要

**问题**：在 H20 96GB 单卡上，MoE Block 基线实现的 SharedExpert 中间激活呈 O(T×I) 增长，T=131K 时高达 6.4 GB；稠密路由表额外占用约 1 GB。长序列训练峰值显存成为瓶颈。

**解决方案**：通过两层正交优化，在数学不改变模型输出的前提下降低峰值显存：

1. **Plan B — Sort-based routing**：用 `sort + bincount` 替代 `F.one_hot`，消除 `[T, K, E]` int64 稠密路由表。T=131K 时节省约 1 GB。
2. **Plan A — Chunked + checkpointed SharedExpert**：将 SharedExpert 前向沿序列维度分块，PyTorch checkpoint 丢弃中间激活后重算。T=131K 时中间激活从 6.4 GB 降至 160 MB。

**核心结果**（H20 96GB 实测）：

| 指标 | 基线 | 优化后 | 改善 |
|------|------|--------|------|
| T=8K 峰值显存 | 4,348 MB | 4,349 MB | 0% |
| T=32K 峰值显存 | 9,376 MB | 7,840 MB | **16.4%** |
| T=131K 峰值显存 | 33,639 MB | 27,495 MB | **18.3%** |
| T=262K 峰值显存 | 64,966 MB | 52,678 MB | **18.9%** |
| T=131K 速度 | 557 ms | 665 ms | +19.4% |
| T=32K 速度 | 152 ms | 177 ms | +16.6% |
| T=2K 速度 | 22.0 ms | 21.2 ms | **-3.7%（更快）** |

**评分预估**（赛题权重 显存 60% + 速度 40%）：

| 序列长度 | 得分 |
|----------|------|
| T=32K | **1.06×** |
| T=131K | **1.07×** |

## 第 1 部分：方案设计

### 1.1 基线架构分析

`MoEBlockBaseline` 前向流程：

```
hidden_states [T, H] → Router(gate) → [T, K] top-k indices/weights
                    → RoutedExperts(one_hot dispatch) → routed_output [T, H]
                    → SharedExpert(hidden_states) → shared_output [T, H]
                    → Add → PostNorm(RMS) → output [T, H]
```

| 组件 | 参数 | 说明 |
|------|------|------|
| MoERouter | `weight [128, 2048]` | Top-K 路由，K=8 |
| MoEExperts | `gate_up_proj [128, 1536, 2048]` + `down_proj [128, 2048, 768]` | 128 个 expert |
| SharedExpert | `gate/up/down Linear(2048, 6144)` | 共享 expert |
| RMSNorm | `weight [2048]` | 最终归一化 |

**基线的三个显存瓶颈**：

| 瓶颈 | 来源 | T=8K 占用 | T=131K 占用 | 严重度 |
|------|------|-----------|-------------|--------|
| 稠密路由表 | `F.one_hot` → `[T, K, E] int64` | ~130 MB | ~1 GB | 高 |
| SharedExpert 中间激活 | gate/up/act/down 全部保留 | ~375 MB | ~6.4 GB | 致命 |
| Expert 中间激活 | 每个 expert 内部线性层 | 中等 | 高 | 中 |

### 1.2 Plan B — Sort-based Routing

#### 原理

Baseline 使用 `F.one_hot(top_k_indices, num_classes=E)` 生成 `[T, K, E]` int64 稠密张量，T=131K 时占用 1.0 GB。优化版本用排序 + 区间访问替代：

```
Step 1: top_k_indices [T, K] 展平为 [T×K]，构造 token 索引 [T×K]
Step 2: 按 expert_idx 排序 → 同一 expert 的 tokens 聚为连续区间
Step 3: bincount 统计每个 expert 的 token 数 → cumsum 得偏移量
Step 4: 直接区间切分 sorted_token[start:end] 访问每个 expert 的 tokens
```

核心代码对比：

```python
# Baseline: 稠密路由表
expert_mask = F.one_hot(top_k_indices, num_classes=E)      # [T, K, E] int64 ~1 GB
expert_mask = expert_mask.permute(2, 1, 0)                 # [E, K, T]
top_k_pos, token_idx = torch.where(expert_mask[expert_idx])

# Optimized: 排序 + 区间访问
sorted_expert, sort_order = expert_flat.sort()             # [T×K]
sorted_token = token_idx[sort_order]                       # [T×K]
expert_counts = torch.bincount(sorted_expert, minlength=E) # [E]
expert_offsets[1:] = expert_counts.cumsum(0)               # [E+1]
t_idx = sorted_token[start:end]                            # 直接区间切分
```

#### 效果

| 序列长度 | Baseline (one_hot) | Optimized (sort) | 节省 |
|----------|-------------------|------------------|------|
| T=2K | 16 MB | <1 MB | ~15 MB |
| T=8K | 130 MB | <2 MB | ~128 MB |
| T=32K | 260 MB | <5 MB | ~255 MB |
| T=131K | 1,073 MB | <10 MB | ~1,063 MB |

#### 数学等价性

两种方法产生的 `(token, expert)` 对应关系完全一致：
- Baseline 通过 one_hot → permute → where 查找
- Optimized 通过 sort → offset 区间访问
- Expert 内对每个 token 使用相同的 `gate_up_proj[e]`/`down_proj[e]` 权重、`top_k_weights` 系数、`index_add_` 聚合方式

### 1.3 Plan A — Chunked + Checkpointed SharedExpert

#### 原理

Baseline 的 SharedExpert 一次性计算所有 T 个 token，autograd 保留全部中间激活：

```
4 个 [T, 6144] bf16 张量 = 4 × T × 6144 × 2 bytes
T=131K: 4 × 131072 × 6144 × 2 = 6.44 GB
```

优化策略：沿序列维度分块，每块通过 `torch.utils.checkpoint.checkpoint` 丢弃中间激活，反向时重算。

```
chunk_size = 40 MB / (6144 × 2 bytes) ≈ 3,413 tokens
每块中间激活: 4 × 3,413 × 6144 × 2 ≈ 160 MB
T ≤ chunk_size 时跳过 checkpoint（无开销）
```

显存量级变化：**6.4 GB → 160 MB**（40× 压缩）。

#### 关键参数

```python
# 最终版参数（H20 实测调优后）
chunk_size = (40 * 1024 * 1024) // (self.intermediate_size * 2)  # ≈ 3413
use_reentrant = True           # 比 False 快 30-50%
T <= chunk_size → 直接计算    # 短序列零开销
```

#### 时空权衡

Checkpointing 代价是反向时重算前向。调优前（chunk_size=341, use_reentrant=False）T=131K 下 +73.4% 的速度损失，优化后（chunk_size=3413, use_reentrant=True）降至 **+19.4%**。

### 1.4 两层优化的组合效果

Plan B 和 Plan A 显存节省正交叠加：

```
总显存节省 ≈ Plan B(路由表 ~1 GB) + Plan A(SharedExpert 激活 ~6 GB)
           ≈ 7 GB @ T=131K（实测 6.1 GB，差异来自其他固定开销）
```

不优化 RoutedExperts 中间激活的原因：每个 expert 仅处理 ~T×K/E 个 token，中间激活量级远小于 SharedExpert，checkpoint 重算开销得不偿失。

| | Baseline | Optimized |

96GB 环境上两者均能运行到 262K。优化的核心价值在于提供约 12 GB 额外显存余量，为优化器状态（Adam 约需 2.4 GB）和其他训练组件留出空间。

### 2.4 与理论预期的符合度

| 指标 | 理论预测 | 实测 | 符合度 |

速度损失略高于预期的主要原因是 checkpoint 重算的串行化效应——逆传播期间的重算无法与 RoutedExperts 的反向叠加执行。

## 第 6 部分：总结与展望

### 关键发现

1. **SharedExpert 激活是真正瓶颈**，T=131K 达 6.4 GB，路由表仅 1 GB
2. **checkpoint 分块大小对速度影响巨大**，341→3413 提升 10× 后速度损失从 73% 降至 19%
3. **短序列跳过 checkpoint 是纯收益**，T≤3.4K 时逻辑自动短路，反而比基线快
4. **use_reentrant=True 是正确选择**，在 PyTorch 2.8 中无兼容性问题

### 局限性

- 单卡优化，未利用 expert 并行或 tensor 并行
- 未使用自定义 CUDA/Triton kernel
- SharedExpert 重算与 RoutedExperts 无法叠行，存在串行化损失

### 后续改进方向

- **torch.compile**：对 `_shared_expert_chunk` 应用编译优化，进一步降低重算开销
- **gate/up 融合**：合并为单个 Linear(2048, 12288)，减少一半 kernel launch
- **自定义 kernel**：编写 sort-based routing 的 fused Triton kernel
