## 🚨 常见问题排查

### Q1: "CUDA out of memory" 立即出现
**原因**：GPU 内存被其他进程占用

**解决方案**：
```bash
# 清空 GPU 缓存
python3 -c "import torch; torch.cuda.empty_cache()"

# 或者重启 CUDA
nvidia-smi # 查看占用
pkill -f python # 强制杀死所有 Python 进程

# 或者临时使用 fp32 替代 bf16（更耗内存，用于诊断）
python3 test_gpu_performance.py --dtype fp32
```

### Q2: 性能波动很大（std 太大）
**原因**：系统负载波动，内存碎片化

**解决方案**：
```bash
# 增加预热步数
python3 test_gpu_performance.py --warmup 10 --measure 30

# 或者关闭系统其他服务
# Linux: systemctl stop docker（如果在用）
# 或者运行 nvidia-smi -pm 1 启用持久模式
nvidia-smi -pm 1
```

### Q3: Optimized 没有比 Baseline 快
**原因**：checkpoint 重计算开销大，或 chunk_size 设置不合理

**解决方案**：
1. 检查 chunk_size 设置（在 solution.py 中）
   ```python
   # 当前设置
   chunk_size = max(1, (4 * 1024 * 1024) // (self.intermediate_size * 2))
   
   # 可以尝试：
   # chunk_size = 512  # 更大的块，更少的重计算
   # chunk_size = 256  # 更小的块，更少的激活显存
   ```

2. 检查 use_reentrant 设置
   ```python
   # 当前（推荐）
   checkpoint.checkpoint(..., use_reentrant=False)
   
   # 可以试试
   checkpoint.checkpoint(..., use_reentrant=True)  # 更快但不支持嵌套
   ```

### Q4: Baseline 在很小的序列长度就 OOM
**原因**：GPU 内存不足或其他进程占用

**解决方案**：
```bash
# 检查可用内存
nvidia-smi

# 用 fp16 代替 bf16（省显存）
python3 test_oom_boundary.py --dtype fp16

# 或者在更新的 GPU 上运行
```

### Q5: 运行脚本报 "No module named 'baseline'"
**原因**：不在正确的目录

**解决方案**：
```bash
# 确保在 附件 目录
cd 附件

# 检查文件是否存在
ls baseline.py solution.py

# 然后再运行
python3 test_gpu_performance.py ...
```

## 🎓 技术深度解析

### Baseline 中的性能瓶颈

```python
# 问题 1: [T, K, E] int64 稠密路由表
expert_mask = F.one_hot(top_k_indices, num_classes=self.num_experts)
# 形状: [T, K, E] = [131072, 8, 128]
# 显存: 131072 * 8 * 128 * 8 bytes = 1 GB!

# 问题 2: SharedExpert 中间激活完全保留
def forward(self, x):
    # x: [T, H] = [131072, 2048]
    gate_out = self.gate_proj(x)      # [T, 6144]
    up_out = self.up_proj(x)          # [T, 6144]
    # 中间激活: ~6 GB @ T=131K
    hidden = F.silu(gate_out) * up_out
    return self.down_proj(hidden)
```

### Optimized 中的优化策略

```python
# 优化 1: sort-based routing
sorted_expert, sort_order = expert_flat.sort()
expert_counts = torch.bincount(sorted_expert, minlength=E)
# 消除 [T, K, E] int64 表，改用 [T*K] 索引 + [E] 计数
# 节省: ~1 GB @ T=131K

# 优化 2: checkpointed shared expert
for chunk in chunks:  # chunk_size ~ 256-512 tokens
    chunk_out = checkpoint.checkpoint(
        shared_expert_chunk, chunk_x
    )
# 前向: 计算激活 → checkpoint 存储 (占用小)
# 反向: 从 checkpoint 重新计算激活 (时间增加但显存省)
# 节省: ~6 GB @ T=131K，代价 +3-7% 耗时
```

### 为什么 Optimized 整体更快？

即使 checkpoint 会增加反向的计算，但通常整体性能会有所改善的原因：

1. **内存访问更快**：sort-based 路由使同一 expert 的 tokens 连续，提高 cache 局部性
2. **减少显存压力**：更小的峰值显存意味着 GPU 不需要 swap 或降频
3. **并行度提升**：更小的中间激活允许 GPU 核心更高效地并行处理
